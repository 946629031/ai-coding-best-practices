## 一、为什么 AI 会写烂代码 / 产生幻觉？

理解问题根源，是解决问题的第一步。

### 1️⃣ AI 默认只覆盖「正常路径（Happy Path）」

AI 生成的代码很少处理：

- `null` / `undefined`
- 边界值
- 并发 / race condition

**结果**：代码"看起来对"，但上线后会出问题。

---

### 2️⃣ 上下文不完整（最大杀手）

AI 看不到完整项目，很可能：

- 改了函数签名但漏改调用处
- 引入不存在的依赖
- 使用过时 API

**结果**：能跑，但集成就崩。

---

### 3️⃣ 幻觉 = "合理但错误"

AI 会：

- 编造 npm 包（甚至存在安全风险）
- 编造 API 接口
- 编造库的用法

> ⚠️ 这类问题最危险，因为错误不明显，容易被直接使用。

---

### 4️⃣ 人对 AI 过度信任

这是最真实、也最常见的问题：

> **最大的坑，就是直接相信 AI 的输出。**

---

## 二、核心方法论（10 条工程级策略）

### 🧠 方法 1：把 AI 当「实习生」，不是「专家」

建立正确的心智模型：

> **AI ≈ 初级工程师（写得快，但不靠谱）**

| ❌ 不要做 | ✅ 要做 |
|---|---|
| 直接复制运行 AI 代码 | code review + 理解逻辑后再运行 |

---

### 🧪 方法 2：测试驱动 AI（最有效策略）

这是目前最强策略，没有之一。

**正确流程：**

```
1. 让 AI 先写测试（unit test）
2. 再让 AI 写实现代码
3. 跑测试
4. 让 AI 修复失败用例
```

**为什么有效：**

- AI 会"对着标准修正自己"，形成闭环
- 测试是 AI 自我纠错的关键机制

---

### 🧾 方法 3：强约束 Prompt（非常关键）

**❌ 错误方式：**

```
帮我写一个登录接口
```

**✅ 正确方式：**

```
技术栈：Node + Express + TypeScript

要求：
1. 必须包含输入校验（使用 zod）
2. 必须处理所有异常情况
3. 返回结构统一（code / message / data）
4. 写对应单元测试
5. 不允许使用任何未说明的依赖

输出：
- 完整实现代码
- 对应测试代码
```

**原理**：限制 AI 的自由度 = 降低幻觉的空间。

---

### 🧱 方法 4：固定「项目上下文」文件

每次对话将项目上下文文件喂给 AI，减少 AI 猜测，从源头避免幻觉。

> 👉 参考本文档 [第五节](#五项目上下文文件模板) 的模板。

---

### 🔄 方法 5：定期"清空上下文"

AI 会被"历史错误污染"（业内称：**context pollution / 上下文污染**）。

**表现**：一开始答错 → 后面一直错

**解决方案：**

- 每完成一个任务 → 开新对话
- 或明确说：「忽略之前的上下文，重新开始」

---

### 🔍 方法 6：双 AI 交叉审查（高手用法）

类似 code review 机制，明显减少 bug。

**流程：**

```
Claude 写代码
  → Codex 审查
    → Claude 修复
      → 循环迭代
```

---

### 🛡 方法 7：依赖必须手动验证（防"假包"）

**AI 常见错误：**

```js
import { something } from "super-utils-lib" // ❌ 这个包根本不存在
```

**你必须：**

- 遇到不熟悉的包 → 去 [npmjs.com](https://www.npmjs.com) 或官方文档验证
- 不认识的包，一律保持怀疑

> ⚠️ 使用不存在或伪造的包，可能遭遇**供应链攻击**。

---

### ⚙️ 方法 8：让 AI 解释代码

用这个 Prompt 验证 AI 的输出质量：

```
逐行解释这段代码，并指出所有潜在的 bug 和边界情况。
```

> 如果 AI 解释不清楚 = 代码有问题，或者它在瞎编。

---

### 🧩 方法 9：小步拆任务（极重要）

**❌ 错误：**

```
帮我写一个完整的电商系统
```

**✅ 正确：**

```
1. 先设计接口（API 设计）
2. 再写用户登录模块
3. 再写商品列表模块
4. 再写支付模块
```

**原因：**

- AI 擅长**局部**实现
- AI 不擅长**系统级**设计

---

### 🧠 方法 10：关键逻辑一定人工介入

以下场景**不要完全依赖 AI**，必须人工审查甚至手写：

| 场景 | 原因 |
|---|---|
| 权限系统 | 错误即安全漏洞 |
| 金融 / 计费逻辑 | 精度与边界要求极高 |
| 并发处理 | AI 对 race condition 不敏感 |
| 安全相关代码 | 幻觉的代价无法接受 |

## 三、推荐开发流程（可直接照做）

```
┌─────────────────────────────────────────┐
│  Step 1  写清楚需求 + 技术约束            │
│  Step 2  让 AI 先写测试用例              │
│  Step 3  让 AI 写实现代码               │
│  Step 4  本地运行，执行测试              │
│  Step 5  让另一个 AI（或同事）review     │
│  Step 6  人工 review（重点看边界处理）   │
│  Step 7  确认无误 → 合入代码             │
└─────────────────────────────────────────┘
```

> ✅ 按此流程执行，可以显著减少 AI 引入的 bug 和幻觉风险。

## 四、前端项目 Prompt 模板

适用于 Vue3 / React 项目的强约束 Prompt 模板：

```
【任务】
[描述你要实现的功能，一句话说清楚]

【技术栈】
- 框架：Vue 3 / React 18
- 语言：TypeScript（严格模式）
- 打包：Vite
- HTTP：Axios（统一封装）
- 状态管理：Pinia / Zustand

【代码规范】
- 禁止使用 any 类型
- 使用 Composition API（Vue）/ Hooks（React）
- 组件单一职责，超过 200 行需拆分
- 所有异步操作必须处理 loading / error 状态

【必须包含】
1. 完整的 TypeScript 类型定义
2. 异常处理（try/catch + 用户提示）
3. 边界情况处理（空数据、加载中、错误状态）
4. 对应单元测试（Vitest）

【禁止事项】
- 不允许引入未在技术栈中说明的依赖
- 不允许使用 console.log（用专用 logger）
- 不允许出现魔法数字（使用常量）

【输出格式】
- 实现代码（带注释）
- 测试代码
- 简要说明实现思路
```

## 五、项目上下文文件模板

在项目根目录创建 `project-context.md`，每次对话时将其内容喂给 AI：

```markdown
# project-context.md

## 项目基本信息
- 项目名：[你的项目名]
- 类型：[前端 / 后端 / 全栈]
- 阶段：[开发 / 维护]

## 技术栈
- 前端：Vue 3 + TypeScript + Vite
- 状态管理：Pinia
- HTTP：Axios（统一封装在 src/utils/request.ts）
- UI 组件库：Element Plus
- 测试：Vitest + Vue Test Utils

## 代码规范
- 禁止使用 any
- 使用 Composition API
- 所有接口返回统一格式：{ code, message, data }
- 错误处理统一通过 src/utils/error.ts

## 目录结构
src/
├── api/        # 接口定义
├── components/ # 公共组件
├── composables/ # 组合式函数
├── stores/     # Pinia 状态
├── utils/      # 工具函数
└── views/      # 页面

## 已有工具函数（禁止重复实现）
- useRequest()：封装异步请求
- useTable()：封装表格分页
- formatDate()：日期格式化

## 禁止事项
- 不允许引入 project-context.md 未列出的 npm 包
- 不允许修改 src/utils/ 下的基础工具（需提前说明）
```
